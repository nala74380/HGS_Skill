# 终端用户体验审查协议

## 发布版装配位置

- 运行层级：`protocols/41_P8_EndUser_Experience_Protocol.md`
- 由 `00_HGS_Master_Loader.md` 统一装配
- 本文件负责体验 / 再审协议，不单独承担角色执行

终端用户是系统的最终消费者。
他们不懂技术，他们只知道"能不能用"。
他们的挫败感，是产品最诚实的评分。

工程师看不到的东西，用户天天在感受：
- 哪个错误提示看不懂
- 哪个步骤找不到入口
- 哪个操作做了三次还没成功
- 哪个问题找代理也解决不了

**这个协议把用户的挫败感，转化成产品可以改进的具体问题。**

## 终端用户审查的六个维度

### 维度一：首次使用成功率审查

```
核心问题：新用户拿到账号，自己能成功登录吗？

关键路径：收到账号密码 → 下载客户端 → 首次登录 → 首次激活

每个节点的卡点记录：

  节点1：下载客户端
    □ 用户能找到下载地址吗？
    □ 下载说明是否清晰？（PC Console / 懒人精灵 分别怎么下载）
    □ 安装过程有没有报错/拦截？
    卡点记录：[具体卡点] + [发生频率]

  节点2：首次登录
    □ 登录界面是否直观？（用户名/密码输入框明确？）
    □ 首次登录需要几步？
    □ 登录失败的提示够不够清晰？（密码错误 vs 账号不存在 vs 账号到期）
    卡点记录：[具体卡点] + [发生频率]

  节点3：首次激活（Worker/懒人精灵）
    □ 用户理解"激活名额"的概念吗？
    □ 激活成功后有明确的成功提示吗？
    □ 名额不足时的提示，用户能理解怎么处理吗？
    卡点记录：[具体卡点] + [发生频率]

首次使用成功率 = 能独立完成到"首次成功运行"的用户比例
  估算：记录最近 10 个新用户，有几个需要代理帮助才能完成首次登录？
  ≤ 2个需要帮助 → 良好（绿）
  3-5个需要帮助 → 有改进空间（黄）
  ≥ 6个需要帮助 → 首次体验有严重问题（红）
```

### 维度二：错误信息可理解度审查

```
核心问题：用户看到报错，能不能自己判断怎么处理？

对每个常见错误，记录用户实际的反应：

  错误：账号到期
    □ 用户看到提示后，知道要"联系代理续期"吗？
    当前提示文案：[记录实际显示的文案]
    用户实际反应：[他们以为发生了什么]
    评分：清晰 ✓ / 模糊 △ / 看不懂 ✗

  错误：激活名额已满
    □ 用户知道"名额"是什么意思吗？
    □ 用户知道下一步该做什么吗？（解绑旧设备 vs 联系代理加购）
    当前提示文案：[记录实际显示的文案]
    用户实际反应：[他们以为发生了什么]
    评分：清晰 ✓ / 模糊 △ / 看不懂 ✗

  错误：无权限访问该项目
    □ 用户知道是"没有买这个项目"还是"账号出了问题"吗？
    当前提示文案：[记录实际显示的文案]
    用户实际反应：[他们以为发生了什么]
    评分：清晰 ✓ / 模糊 △ / 看不懂 ✗

  错误：设备不匹配
    □ 用户知道这是换手机/重装导致的吗？
    当前提示文案：[记录实际显示的文案]
    用户实际反应：[他们以为发生了什么]
    评分：清晰 ✓ / 模糊 △ / 看不懂 ✗

  错误：版本需要更新
    □ 用户知道在哪里更新吗？
    当前提示文案：[记录实际显示的文案]
    用户实际反应：[他们以为发生了什么]
    评分：清晰 ✓ / 模糊 △ / 看不懂 ✗

汇总：
  "看不懂 ✗" 的数量 = 需要重写的错误提示数量
  → 直接传递给产品层改进文案
```

### 维度三：高频投诉模式审查

```
核心问题：用户重复抱怨的是什么？

统计最近 30 天用户主动反映的问题：

  用户原话（不翻译，保留原始语言）：
  | 用户说的话 | 出现次数 | 实际根因 | 是产品问题还是用户操作问题 |
  |-----------|---------|---------|--------------------------|
  | "怎么又要重新登录" | ? | token过期 | 产品问题（过期提示不够提前）|
  | "换手机好麻烦" | ? | 换机流程复杂 | 产品问题（缺少一键换机）|
  | "不知道找谁解决" | ? | 联系方式不清晰 | 产品问题（支持渠道不明确）|
  | "说名额满了但我没用那么多" | ? | 废弃设备占名额 | 产品问题（缺少废弃设备提醒）|
  | [其他原话] | ? | ? | ? |

关键洞察：
  出现次数 ≥ 3次 且 根因是产品问题 → 必须列入改进清单
  用户原话中有情绪词（"好麻烦"、"又"、"一直"）→ 是高强度痛点

用户原话的价值：
  不要翻译成技术语言。
  "怎么又要重新登录"比"token刷新机制用户感知差"更有说服力。
  保留原话，让产品经理直接看到用户的真实感受。
```

### 维度四：关键操作完成率审查

```
核心问题：用户能独立完成关键操作吗？

关键操作自助完成率评估：

  操作：换手机（解绑旧设备）
    □ 用户能自己完成吗？还是必须联系代理？
    □ 操作入口容易找到吗？
    □ 操作步骤几步？
    自助完成率估算：[能自己完成的用户比例]

  操作：查看自己有哪些激活设备
    □ 入口在哪里？用户能找到吗？
    □ 展示的信息够不够让用户判断哪台设备可以解绑？
    自助完成率估算：[能自己完成的用户比例]

  操作：查看账号到期时间
    □ 到期时间在哪里显示？用户能主动查到吗？
    □ 快到期时有提醒吗？
    自助完成率估算：[能自己完成的用户比例]

  操作：查看有哪些项目可以用
    □ 项目列表是否清晰展示授权状态？
    □ "可见但不可进入"的项目，用户知道为什么吗？
    自助完成率估算：[能自己完成的用户比例]

汇总：自助完成率 < 50% 的操作 = 需要优化的高优先级体验问题
```

### 维度五：用户求助行为审查

```
核心问题：用户遇到问题时的求助路径合理吗？

  □ 用户遇到问题，第一反应是看错误提示、看帮助文档，
    还是直接找代理/客服？
    → 大部分直接找人 = 错误提示和帮助文档没有发挥作用

  □ FAQ 文档用户会主动看吗？
    → 不看 = FAQ 存在但用户不知道在哪里，或者写得用户看不懂

  □ 联系代理的渠道是否清晰？
    → 用户不知道找谁 = 支持渠道说明缺失

  □ 用户找到帮助后，问题能被解决吗？
    → 解决不了 = 代理的能力/信息/权限不足（见代理审查协议）

  求助链条评估：
    错误提示 → 帮助文档 → 代理 → 平台
    每个节点的"漏出率"（在这个节点没解决，去找下一个）
    漏出率高的节点 = 需要加强的节点
```

### 维度六：情绪温度计

```
核心问题：用户对这个系统的总体情绪是什么？

收集最近用户的原始反馈，不加翻译，不加过滤：

  正面反馈（哪些地方用户觉得好用）：
    [原话1]
    [原话2]
    ...

  负面反馈（哪些地方用户觉得难用/烦）：
    [原话1]
    [原话2]
    ...

  中性困惑（用户不明白某件事是怎么运作的）：
    [原话1]
    [原话2]
    ...

情绪词频统计：
  "麻烦"出现次数：___
  "又"出现次数：___（"又报错"、"又要重新登录"）
  "不知道"出现次数：___
  "为什么"出现次数：___

情绪温度：
  负面词频 < 正面词频 → 基本满意（绿）
  负面 ≈ 正面 → 中性，有改进空间（黄）
  负面 > 正面 → 用户体验有问题（红）
  出现"再也不用"、"太差了"等强烈负面 → 立即上升为P0（红❗）
```

## 用户原声的特殊价值

```
用户原话 vs 技术描述，哪个更有说服力？

技术描述：
"token 刷新机制在多标签页场景下用户感知不友好，
 建议优化刷新时机和提示文案"

用户原声：
"为什么我每天都要重新登录一次？
 有时候做到一半突然就说我没登录了，
 我的操作全没了，很烦的"

用户原声会直接告诉你：
  → 频率（"每天"）
  → 场景（"做到一半"）
  → 损失（"操作全没了"）
  → 情绪（"很烦"）

这些细节是技术描述里没有的。
审查报告里，用户原声必须保留，不能翻译掉。
```

## 审查触发机制

```
自动触发：
  → 每月进行一次全量审查
  → 同一问题被 3 个以上用户反映 → 立即触发专项审查
  → 用户出现强烈负面情绪（"再也不用了"类）→ 立即触发

手动触发：
  "帮我做一次用户体验审查"
  "最近用户投诉比较多，整理一下"
  "新功能上线后用户反馈怎么样"
```

## 与代理审查的协同

```
终端用户体验审查          代理体验审查
      ↓                         ↓
  用户看到什么          代理操作了什么
  用户感受如何          代理遇到什么困难
      ↓                         ↓
      └────────────┬────────────┘
                   ↓
           综合体验报告
                   ↓
      P9（一致性）/ P10（战略）/ 文档更新
```

两个审查协议是互补的：
- 用户体验审查 → 发现前台/交互/文案问题
- 代理体验审查 → 发现后台/流程/权限问题
- 合并分析 → 发现端到端的体验断层（代理以为处理好了，用户还是困惑）

## 与主编排器的协作

本协议在主编排器中承担 **环节1：审查** 与 **环节4：体验回访** 的终端用户侧输入职责。
它不直接修问题，但负责把“用户为什么会卡住”变成 P9 能派单、P10 能定优先级的证据。
