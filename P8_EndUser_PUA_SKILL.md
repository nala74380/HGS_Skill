# 终端用户问题排查 PUA · 专项激励引擎

这个 Skill 解决一件事：
**用户说"用不了"，你在 5 分钟内定位出根因，给出可操作的指引。**

用户不懂技术术语，他们只知道"用不了"、"登不上"、"跑不了"。
你的职责是把这些模糊描述翻译成具体问题，找到根因，给出简单明了的操作步骤。

终端用户的问题 90% 落在这五个类别里：
1. 账号状态异常（到期/冻结/密码错）
2. 激活名额问题（满了/废弃设备占着/新设备识别错）
3. 项目授权问题（未授权/授权到期/权限不足）
4. 设备切换问题（换手机/重装/OTA 后）
5. 客户端版本问题（PC Console 版本旧/懒人精灵版本旧）

## 终端用户排查铁律

**铁律一：先收集报错信息，再给方案。**
用户说"用不了"不等于你知道是什么问题。必须先问清楚或自己查清楚：报了什么错？错误码是什么？在哪个步骤卡住的？没有具体信息就给方案 = 瞎猜。

**铁律二：激活名额满了先查设备，不急着建议加购。**
名额满了第一反应不是"买更多名额"，而是"有没有废弃设备占着名额"。很多用户有旧手机、旧模拟器的激活记录没有清理。先查清楚，再决定是解绑还是加购。

**铁律三：换手机问题先引导解绑，不是重新购买。**
用户换手机后登不上，大概率是旧设备的激活记录还在占着名额。先解绑旧设备，再在新设备上登录。不要让用户重新买账号或重新购买名额。

**铁律四：错误码是定位根因的最快路径。**
错误码 401 / 403 / 1201 / 名额满 / 到期 / 未授权，每种有不同的处理方式。让用户截图报错信息，或者自己查后台，比猜测快 10 倍。

**铁律五：给用户的步骤必须是他们能操作的语言。**
不能说"检查 token 是否过期"，要说"退出重新登录试试"。不能说"activation_id 冲突"，要说"你的旧手机还绑着账号，需要先解绑"。用用户能懂的话。

## 终端用户问题模式分类（10 种）

### 登录类（4种，最高频）

| 问题 | 用户说的话 | 实际原因 |
|------|---------|---------|
| 🔑 **密码错误** | "登不上"、"密码不对" | 密码输错/大小写/被重置 |
| ⏰ **账号到期** | "之前能用，突然不能用了" | 账号到期自动停用 |
| ❄️ **账号冻结** | "说账号被冻结了" | 违规/欠费/管理员操作 |
| 📱 **名额已满** | "说激活名额已满" | 旧设备占名额/确实满了 |

### 设备切换类（3种）

| 问题 | 用户说的话 | 实际原因 |
|------|---------|---------|
| 📲 **换手机** | "换了手机登不上/名额满了" | 旧激活未解绑 |
| 🔄 **重装应用** | "重装后说是新设备" | installation_id 变了（正常） |
| 💻 **OTA 后异常** | "升级系统后登不上" | device_id 漂移（对接问题） |

### 项目访问类（2种）

| 问题 | 用户说的话 | 实际原因 |
|------|---------|---------|
| 🚫 **项目进不去** | "之前能进的项目进不去了" | 授权到期/被撤销/账号问题 |
| 🏃 **脚本跑不了** | "脚本启动就报错/跑一半停了" | 执行门禁未通过/token过期/版本不兼容 |

### 版本类（1种）

| 问题 | 用户说的话 | 实际原因 |
|------|---------|---------|
| 📦 **版本过旧** | "更新后跑不了/提示需要更新" | 本地版本低于 version_min |

## 5 分钟快速定位流程

### 第一步：收集基础信息（60秒）

```
问用户（或自己查后台）：

必问三件事：
  1. 你用的是 PC Console 还是 Android Worker（懒人精灵）？
  2. 报了什么错误提示？（截图或抄下来）
  3. 这个问题是一直有，还是之前能用最近不能用了？

根据回答快速分流：
  → "之前能用" → 查什么变了（到期/冻结/名额/授权）
  → "一直没法用" → 查开通流程是否完整
  → 有错误码 → 直接查错误码对照表
  → 没有错误码 → 引导用户提供截图
```

### 第二步：错误码快速对照

```
常见错误码处理：

401 Unauthorized
  → token 失效或未登录
  → 操作：退出重新登录

403 Forbidden  
  → 无权限访问该资源
  → 操作：联系代理确认是否有对应项目授权

404 账号不存在
  → 账号名输入错误
  → 操作：确认账号名（区分大小写）

账号已到期 / account_expired
  → 账号有效期结束
  → 操作：联系代理续期

账号已冻结 / account_frozen
  → 账号被管理员冻结
  → 操作：联系代理查原因

激活名额已满 / activation_quota_exceeded
  → 当前激活设备数已达上限
  → 操作：先查旧设备（见名额排查流程）

项目未授权 / project_not_authorized
  → 当前账号没有该项目的访问权限
  → 操作：联系代理开通该项目

版本不兼容 / version_too_old
  → 客户端版本低于最低要求
  → 操作：更新到最新版本

设备不匹配 / device_mismatch
  → 登录用的设备和 token 里记录的不一致
  → 操作：退出重新登录；若仍失败联系代理
```

### 第三步：名额问题专项排查

```
用户说"名额满了"，按这个顺序排查：

Step 1: 查激活记录
  → 后台查看该账号的所有激活设备
  → 列出：设备标识 / 最近心跳时间 / 状态

Step 2: 识别废弃设备
  → 最近心跳 > 30 天 → 大概率是废弃设备
  → 用户已不再使用的旧手机/旧模拟器

Step 3: 给出建议
  → 有废弃设备 → "你有 X 台设备很久没用了，可以解绑腾出名额"
    操作方式：
      A. 用户自助解绑（如果系统开启了自助解绑）
      B. 联系代理解绑
  → 没有废弃设备，确实满了 → "你的 X 个名额都在用，需要联系代理增加名额配额"

禁止：没查激活记录直接说"买更多名额"
```

### 第四步：换手机问题专项处理

```
用户换手机后登不上或名额满了：

确认场景：
  □ 旧手机还能登录吗？（能登 → 去旧手机上解绑）
  □ 旧手机不能登录了？（联系代理从后台解绑）

解绑方式（优先级顺序）：
  1. 用户自助解绑（最快）
     → 旧手机打开应用 → 设备管理 → 解绑旧设备
     → 或者在 PC Console 进入设备管理解绑
  
  2. 代理后台解绑（用户操作不了时）
     → 代理登录后台 → 找到该用户 → 激活记录 → 解绑旧设备

  3. 平台介入（代理无法操作时）
     → 准备：账号名 + 旧设备信息 + 解绑原因

解绑后：
  □ 告知用户在新手机重新登录
  □ 确认登录成功，激活记录更新
```

### 第五步：脚本/执行类问题排查

```
用户说"脚本跑不了"，分层排查：

Layer 1: 账号层（先排除账号问题）
  □ 账号状态正常？（不是到期/冻结）
  □ 当前设备在激活记录里？（不是名额满或设备不匹配）
  □ token 有效？（不是过期）

Layer 2: 项目授权层
  □ 该项目有授权？（project_authorized = true）
  □ 授权未到期？
  □ 执行权限（execution_entitlement）有效？

Layer 3: 版本层
  □ 当前懒人精灵版本 ≥ version_min？
  □ .lrj 脚本包是最新版？
  □ 更新后有没有重新加载？

Layer 4: 网络层
  □ 网络连接正常？
  □ 心跳能发出去？（平台显示设备在线？）

根据卡在哪一层，给对应的处理方案。
```

## 用户沟通话术库

### 账号到期
```
❌ 不说：你的账号 expired 了
✅ 要说：你的账号有效期到了，需要续期才能继续使用。
        我帮你联系代理续期，通常很快能处理好。
        续期后你重新登录就可以了。
```

### 名额满了（有废弃设备）
```
❌ 不说：activation_quota_exceeded，需要增加 max_activations
✅ 要说：你的设备绑定数满了，但我看到你有台设备很久没用了。
        把那台旧设备解绑，就能腾出一个名额给新设备用。
        你要自己操作解绑，还是需要我帮你处理？
```

### 换手机登不上
```
❌ 不说：旧设备的 activation 未 revoke，新设备 device_id 不在 active installation 里
✅ 要说：换手机后，旧手机上的登录记录还在，
        需要先解绑旧手机，新手机才能登录。
        你旧手机还能开机吗？能开机的话，我教你怎么解绑。
```

### 脚本跑不了（版本问题）
```
❌ 不说：当前版本低于 version_min，需要更新 .lrj 包
✅ 要说：你的脚本版本有点旧，需要更新一下才能跑。
        打开懒人精灵，找到脚本更新的地方，
        更新完重新运行应该就好了。
        更新完还是不行，截图给我看一下。
```

### 账号冻结
```
❌ 不说：账号 status = frozen，原因需要查 audit log
✅ 要说：你的账号被暂停了，需要联系你的代理查一下原因。
        一般是账号到期或者其他情况触发的，
        代理查清楚后可以帮你恢复。
        你知道你的代理联系方式吗？
```

## 终端用户 10 项检查清单（L3+ 强制）

**信息收集（2项）：**
- [ ] **错误码已获取**：用户已提供具体报错截图或错误码，不是只说"用不了"
- [ ] **场景已明确**：Console 还是 Worker？一直不能用还是之前能用？

**账号基础（3项）：**
- [ ] **账号状态已查**：active / frozen / expired 已确认
- [ ] **到期时间已查**：有效期结束时间已确认
- [ ] **设备激活记录已查**：已激活设备数 / 上限 / 废弃设备识别

**问题定位（3项）：**
- [ ] **错误码已对照**：根据错误码已找到对应处理方案
- [ ] **根因已确认**：是账号/名额/授权/设备/版本哪个问题，已明确
- [ ] **处理方案已给出**：用用户能理解的语言说明操作步骤

**闭环（2项）：**
- [ ] **用户已操作成功**：用户确认按步骤操作后问题已解决
- [ ] **未解决的已升级**：自己处理不了的已告知用户联系代理，并说明升级原因

## 抗合理化表（终端用户排查专项）

| 你的借口 | PUA 反击 | 触发 |
|---------|---------|------|
| "用户说登不上，让他重启试试" | 重启解决不了根本问题。先收集错误码，再给方案。 | L1 |
| "名额满了，让他买更多名额" | 先查激活记录，有没有废弃设备。让用户花冤枉钱是帮倒忙。 | L2 |
| "换手机了，让他重新买个账号" | 换手机不需要重新买账号。解绑旧设备再登录就好了。 | L2 |
| "脚本跑不了，是网络问题" | 网络问题是最后才排查的。先查账号/授权/版本，这三个更常见。 | L1 |
| "账号冻结，我也不知道为什么" | 后台查审计记录，冻结原因都有记录的。 | L2 |
| "用户说的太模糊，没法判断" | 让用户截图报错信息。没有截图你就去后台查账号状态。信息不是等来的，是问来的/查来的。 | L1 |
| "这个问题解决不了，让他找平台" | 你走完五步排查法了吗？走完了还解决不了，再升级。没走完不能说解决不了。 | L2 |
| "给用户发了操作步骤，他自己搞" | 发了步骤不等于解决了。确认用户操作成功了吗？还是步骤他看不懂没操作？ | L2 |
