# 代理体验审查协议

代理是系统和终端用户之间的中间层。
他们每天重复操作相同流程，踩相同的坑。
他们的痛点不是个人技能问题，而是**产品设计信号**。

这个协议做一件事：
**把代理的日常摩擦，转化为可以推动产品改进的结构化证据。**

---

## 代理审查的五个维度

### 维度一：操作效率审查

```
核心问题：完成一次常规操作需要几步？每步合理吗？

审查项目：
  □ 开一个新客户：从进入后台到完成确认，实际走了几步？
    期望值：≤ 5步
    记录：[实际步骤数] + [哪步多余/哪步缺少信息]

  □ 为客户续期：找到客户→确认到期时间→完成续期，几步？
    期望值：≤ 3步
    记录：[实际步骤数] + [卡点在哪里]

  □ 为客户授权一个项目：查余额→授权→确认，几步？
    期望值：≤ 4步
    记录：[实际步骤数] + [最容易出错的步骤]

  □ 解绑客户旧设备：找到客户→找到激活记录→解绑，几步？
    期望值：≤ 4步
    记录：[实际步骤数] + [最耗时的步骤]

评分标准：
  所有操作都在期望值内 → 效率良好（绿）
  有 1-2 项超出期望值 → 有优化空间（黄）
  3 项以上超出期望值 → 流程需要重新设计（红）
```

### 维度二：信息透明度审查

```
核心问题：后台给代理的信息够不够让他自己判断客户问题？

必须能直接看到的信息：
  □ 客户账号状态（active/frozen/expired）✓/✗
  □ 账号到期时间精确到日 ✓/✗
  □ 当前已激活设备数 / 上限 ✓/✗
  □ 每台设备的最近心跳时间 ✓/✗
  □ 当前点数余额 + 冻结额 + 可用额（三个数分开显示）✓/✗
  □ 最近 N 笔流水（扣点原因/对应授权/时间）✓/✗
  □ 客户名下的所有项目授权状态 ✓/✗
  □ 每个授权的到期时间 ✓/✗

发现"✗"的项目：
  → 代理无法自己判断，每次都要找平台 = 信息缺口
  → 记录：[缺失信息名称] + [代理需要这个信息的场景]

信息缺口数量：
  0个 → 信息充分（绿）
  1-2个 → 部分缺失，有改进需求（黄）
  3个以上 → 信息严重不足，代理依赖平台过多（红）
```

### 维度三：高频问题模式审查

```
核心问题：同一类客户问题反复出现，说明什么？

统计最近 30 天客户投诉/咨询，按类型分类：

  | 问题类型 | 发生次数 | 代理能自己解决？ | 根因 |
  |---------|---------|----------------|------|
  | 账号到期 | ? | 是/否 | ? |
  | 名额满了 | ? | 是/否 | ? |
  | 项目进不去 | ? | 是/否 | ? |
  | 换手机问题 | ? | 是/否 | ? |
  | 脚本跑不了 | ? | 是/否 | ? |
  | 密码问题 | ? | 是/否 | ? |
  | 其他 | ? | 是/否 | ? |

分析：
  发生次数最高的 TOP 3 问题 → 是否可以通过产品改进预防？
  代理无法自己解决的问题 → 是信息不足还是权限不足？
  每月重复出现的问题 → 是否需要写入 FAQ 或改进产品流程？
```

### 维度四：点数管理体验审查

```
核心问题：代理管理点数的全流程顺不顺？

  □ 查余额是否一步可达？（不需要多层点击）
  □ 可用额 = 余额 - 冻结额，这个计算是否系统自动显示？
    还是需要代理手动算？
  □ 冻结额产生后，系统会主动通知代理吗？
    还是代理要自己发现？
  □ 申请冲正的流程是几步？能否在后台直接操作？
    还是需要联系平台人工处理？
  □ 历史流水能导出成表格吗？
  □ 充值到账时间和预期是否一致？有延迟时有通知吗？

痛点记录：
  [具体的操作摩擦点] + [期望的改进方式] + [影响频率：每天/每周/每月]
```

### 维度五：异常处理能力审查

```
核心问题：遇到非常规情况，代理有多少能力自己处理？

场景测试：
  场景A：客户账号被冻结，代理能查到冻结原因吗？
    → 能 ✓ / 看不到 ✗ / 只看到"已冻结"无原因 △

  场景B：某笔扣点失败导致冻结，代理能自己申请冲正吗？
    → 可以在后台直接操作 ✓ / 需要联系平台 ✗

  场景C：客户说"我的账号被别人登录了"，代理能查到登录记录吗？
    → 能看到登录IP和时间 ✓ / 看不到 ✗

  场景D：需要批量为多个客户续期，后台支持批量操作吗？
    → 支持 ✓ / 只能逐个操作 ✗

  场景E：客户不在了，需要回收名额和授权，几步能完成？
    → 记录实际步骤

自主处理率 = 能自己处理的场景 / 总场景数
  ≥ 80% → 代理自主能力强（绿）
  60-80% → 有改进空间（黄）
  < 60% → 代理严重依赖平台支持（红）
```

每次审查完成后输出结构化报告，向上传递：

```
[代理体验审查报告]
审查日期：____
审查人（代理ID）：____
审查周期覆盖：____

━━━ 评分总览 ━━━
操作效率：🟢绿/🟡黄/🔴红
信息透明度：🟢绿/🟡黄/🔴红
高频问题：🟢绿/🟡黄/🔴红
点数管理：🟢绿/🟡黄/🔴红
异常处理：🟢绿/🟡黄/🔴红

━━━ 关键发现 ━━━

【P0 产品缺陷】（影响代理日常运营）
  1. [具体问题] → [发生场景] → [影响频率] → [期望改进]
  2. ...

【P1 体验优化】（有摩擦但能绕过）
  1. [具体问题] → [当前绕过方式] → [期望改进]
  2. ...

【P2 增值需求】（没有也能运营，但会更好）
  1. ...

━━━ 高频问题 TOP3 ━━━
  #1 [问题类型]：本月发生 N 次，建议 [产品/文档/流程] 层面解决
  #2 ...
  #3 ...

━━━ 信息缺口清单 ━━━
  缺失信息：[名称]
  需要场景：[什么时候需要]
  影响：[没有这个信息时代理怎么处理]

━━━ 传递方向 ━━━
  → P9 一致性问题：[如有跨模块不一致的地方]
  → P10 战略决策：[影响产品方向的核心发现]
  → 文档更新：[需要同步到FAQ/操作手册的内容]
```

## 审查触发机制

### 自动触发
```
以下情况自动开启审查：

周期性：每月最后一个工作日执行一次全量审查
事件性触发：
  → 同一周内同一问题被 3 个以上客户反映
  → 某个操作流程导致代理操作失败 2 次以上
  → 点数异常冻结出现
  → 代理说出"每次都这样"、"老问题了"、"一直这样"
```

### 手动触发
```
代理主动发起：
  "帮我做一次体验审查"
  "我想记录一下最近遇到的问题"
  "这个流程太麻烦了，帮我整理一下"
```

## 与 PUA 主 Skill 的关系

```
日常运营 → 代理 PUA Skill（排查问题、穷尽执行）
           ↓ 问题积累
周期审查 → 代理审查协议（结构化记录体验痛点）
           ↓ 输出报告
向上传递 → P9（一致性/接口问题）
           P10（产品优先级/战略方向）
           文档更新（FAQ/操作手册）
```

审查协议不替代 PUA Skill，是在 PUA Skill 之上的**体验反馈闭环层**。

---

## 与主编排器的协作

本协议在主编排器中承担 **环节1：审查** 与 **环节4：体验回访** 的代理侧输入职责。
输出必须被主编排器吸收为结构化问题，而不是停留在感受描述。
