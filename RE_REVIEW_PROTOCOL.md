# 再审协议（RE-REVIEW）

再审不是问"好了吗？"
再审是：
- **P9**：这次修复从根本上解决了吗？系统其他地方还有同类风险吗？
- **P10**：这个问题揭示了什么产品方向信息？优先级需要调整吗？

两个视角缺一不可。只有 P9 和 P10 都签字，问题才能真正关闭。

---

## 再审触发条件

```
环节4体验验证报告产生后，无论结果如何，自动触发再审。

三种触发状态：
  状态A：用户/代理确认"好了" → 执行完整再审（P9+P10）
  状态B：用户/代理说"还是有问题" → 先重新派单，再对原修复做P9复盘
  状态C：无法收到用户反馈 → 72小时后强制触发，以工程验证为准
```

## P9 再审：技术层收尾

### P9 再审的三个问题

```
问题1：根因真的消除了吗？

  不是"问题现象消失了"，是"产生这个问题的条件被移除了"

  检查方式：
    □ 修复前的根因描述：____
    □ 修复后，这个根因还存在吗？
      → 还存在（只改了症状）→ 重新派单，标记为"症状治疗不治根"
      → 已消除 → 继续

  举例：
    根因是"幂等键缺失"
    修复是"加了try-catch忽略重复请求"
    → 根因未消除，只是掩盖了症状 → 重新派单

问题2：系统其他地方还有同类风险吗？

  P9 跨域扫描：
    □ 同类代码模式在其他接口存在吗？
    □ 同类设计缺陷在其他模块存在吗？
    □ 这次修复的变更，有没有对其他模块造成新的不一致？

  扫描范围：
    后端改动 → 检查同类接口（写接口全部）
    前端改动 → 检查同类页面/组件
    对接改动 → 检查对应的另一端是否同步
    文档改动 → 检查引用了这部分内容的其他文档

  输出：
    发现同类风险 → 生成新信号ID，进入派单队列（P2优先级）
    无同类风险 → 继续

问题3：这个问题需要固化为模式吗？

  判断标准（满足任意一条）：
    □ 这是第二次出现的同类问题
    □ 根因分析揭示了一个之前没有文档化的设计规则
    □ 修复方案有复用价值（下次遇到类似问题可以直接套用）
    □ 未来AI智能体生成代码时需要遵守这条规则

  如果需要固化：
    [P9-PATTERN-FIXED]
    模式名称：____（简短，可检索）
    根因类型：幂等/事务/权限/状态机/缓存/其他
    信号特征：____（下次遇到这类问题时的早期识别标志）
    标准做法：____（具体实现规范）
    反模式：____（常见的错误做法）
    验收标准：____（怎么确认实现了这个模式）
    写入规范：[对应规范文档 + 章节号]
    更新Skill：[需要更新哪个P8 Skill的排查流程或抗合理化表]
```

### P9 再审输出格式

```
[P9-RE-REVIEW]
信号ID：____
根因消除确认：是/否
  如否：根因仍存在原因：____，重新派单→ [新信号ID]

跨域扫描结果：
  发现同类风险 N 处：
    风险1：[描述] → 新信号ID：____，优先级：____
    风险2：...
  无同类风险：✓

模式固化：
  需要固化：是/否
  如是：[P9-PATTERN-FIXED 内容]
  写入规范文档：[文档名 + 章节]
  更新Skill：[Skill名 + 具体更新内容]

P9 签字：通过/不通过
不通过原因：____
```

## P10 再审：战略层收尾

### P10 再审的三个问题

```
问题1：这个问题揭示了什么产品信息？

  每次真实的用户/代理痛点，都是产品方向的信号。
  P10 不只是确认"修好了"，而是问"这说明了什么"。

  分析维度：
    □ 这个问题的根因，是产品设计决策造成的，还是实现bug？
      → 设计决策问题 → 是否需要调整产品方向？
      → 实现bug → 是否需要加强质量门禁？

    □ 用户/代理为了绕过这个问题，发展出了什么非预期的使用方式？
      → 非预期使用方式 = 产品没有满足的真实需求
      → 是否需要把这个"绕过方式"变成正式功能？

    □ 这个问题的影响范围，是否超出了当初的预期？
      → 超出预期 → 是否需要重新评估相关功能的优先级？

问题2：当前的优先级队列需要调整吗？

  P10 四象限重新扫描：
    □ 本次修复产生的新信号（P9的跨域扫描结果），
      在当前资源约束下，应该排在哪里？
    □ 有没有因为这次问题，让某个之前是P2的问题升级为P1？
    □ 有没有因为这次修复，让某个之前觉得重要的问题变得不那么重要了？

  输出：
    优先级不变 → 继续
    需要调整 → 输出调整建议（附理由）+ 等待你确认

问题3：文档/规范/Skill体系需要更新吗？

  P10 视角的文档价值判断：
    □ 这次问题的解决过程，是否产生了AI智能体未来需要遵守的新约束？
      → 是 → 必须更新对应Skill（不能只靠"记住了"）
    □ FAQ/用户手册/代理手册需要更新吗？
      → 这个问题之前没有文档 → 补充
      → 文档有但用户/代理没看到 → 检查文档可达性
    □ 这次修复有没有改变某个已有规范？
      → 有 → 规范必须同步（先改规范后改代码的原则反向）
```

### P10 再审输出格式

```
[P10-RE-REVIEW]
信号ID：____
产品信息提取：
  根因性质：设计决策/实现bug/两者都有
  产品信号：____（这个问题说明了什么）
  是否需要调整产品方向：是/否
    如是：调整建议：____，需要你确认

优先级调整：
  维持现有队列：是/否
  如需调整：
    升级：[信号ID] P2→P1，原因：____
    降级：[信号ID] P1→P2，原因：____
    新增：[新信号ID]，优先级：____

文档更新任务：
  FAQ更新：是/否 → [具体更新内容]
  用户手册更新：是/否 → [具体章节]
  代理手册更新：是/否 → [具体章节]
  规范文档更新：是/否 → [文档名 + 章节]（AI执行，低成本）

P10 签字：通过/不通过
不通过原因：____（通常意味着需要更大范围的重新评估）
```

## 再审的三种结果

### 结果A：P9+P10 全部通过 → 完全闭环

```
[LOOP-CLOSED]
信号ID：____
P9签字：通过
P10签字：通过
知识条目：[KNOWLEDGE-ENTRY ID]
本轮循环耗时：____
同类风险清单：____条已加入派单队列
优先级调整：____

→ 等待下次触发（周期性/新信号）
```

### 结果B：P9 不通过 → 重新派单

```
[LOOP-REOPEN-P9]
原因：根因未消除 / 发现跨域更大问题
新信号ID：____
新派单卡：[重新生成，包含P9的分析]
→ 重新进入环节2 DISPATCH
→ 注意：失败计数从原信号继承，不重置
```

### 结果C：P10 不通过 → 战略重评

```
[LOOP-REOPEN-P10]
原因：这个修复方向与产品战略不符 / 发现更高优先级问题
P10建议：____
需要你确认：是（P10不通过必须人工决策）
等待输入：你的战略决策
→ 可能导致：撤销当前修复 / 重新定义问题范围 / 调整整体路线图
```

## 与纯净联动版的差异

```
旧再审（只是确认）：
  "问题解决了吗？解决了→关闭，没解决→重来"

新再审（P9+P10双重）：
  P9："根因消除了吗？还有同类风险吗？要固化模式吗？"
  P10："这个问题说明什么？优先级要调整吗？文档要更新吗？"
  联合："生成完整的知识条目，让下次更快"

关键区别：
  旧版：问题解决就结束
  新版：问题解决只是开始，收尾工作（固化、更新、沉淀）
        才是防止下次复发的关键
```
