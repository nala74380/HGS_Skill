> 版本：formal-2026-03-31-r3

> 说明：本文件是**工具治理文档**，不是运行入口；不替代 `MANIFEST` / `Master Loader`，但为正式装配链提供唯一工具调用依据。

# 一、总原则

## 1. 工具不是角色

HGS 中的工具 Skill 只做四件事：

1. 结构化分析  
2. 证据抽取  
3. 差异识别  
4. 结果格式化  

工具 **不能** 做四件事：

1. 替代 Owner 拍板真相  
2. 越权定义业务规则  
3. 宣布已闭环  
4. 绕过 P9 / P10 的裁决链  

一句话：

**工具负责“看清楚”，角色负责“拍板、执行、验证、收口”。**

## 2. 工具调用的基本顺序

```text
先识别真相 Owner / 执行 Owner
  ↓
再选对应工具做结构化分析
  ↓
工具结果进入 issue evidence / verdict / plan
  ↓
角色基于工具结果拍板或执行
```

## 3. 工具结果的默认落点

工具输出不得悬空，必须进入至少一个协议或角色输出：

- `ISSUE-LEDGER`
- `P8-EXEC-REPORT`
- `P9-REVIEW-VERDICT`
- `QA-VALIDATION-PLAN`
- `QA-VERIFICATION-RESULT`
- `BUSINESS-RULE-DECISION`
- `IDENTITY-DECISION`
- `BILLING-VERDICT`
- `RELEASE-CONFIG-VERDICT`
- `SECURITY-RISK-VERDICT`
- `CONTROL-PLANE-VERDICT`
- `EXECUTION-PLANE-VERDICT`
- `SRE-OBSERVABILITY-VERDICT`
- `AGENT-OPS-VERDICT`
- `ENDUSER-SUPPORT-VERDICT`
- `DOCS-KNOWLEDGE-UPDATE`
- `HGS-CLOSEOUT`

# 八、工具越界红线

## 1. 不能替代真相角色
- `70` 不能替代 Product 拍板规则  
- `72/73/74/88/89/90/91` 不能替代 Auth / Execution Plane / Console Runtime 拍板真相  
- `76/77/78` 不能替代 Billing 拍板账务真相  
- `92/98` 不能替代 SRE / Execution Plane 做最终线上归因结论  
- `101` 不能替代 Release / P10 拍板是否接受发布风险  
- `109/110` 不能替代 Security / Risk Owner 或 P10 拍板最终风险容忍度  

## 2. 不能替代验证结论
- `95` 不能等于“已经验证通过”  
- `96` 不能等于“已经全部测完”  
- `107` 不能等于“字段齐了就代表问题解决了”  
- `108` 不能等于“模拟走通就代表真实执行没问题”  

## 3. 不能替代文档治理
- `106` 不能把未拍板争议写进 SOP  
- `106` 不能替代 Docs Owner 做适用范围和受众把关  

## 4. 不能绕过 P9
- 工具发现多 owner 冲突、contract 漂移、状态漂移、错派、运行链断点或绕过风险时，必须回到 P9 拆单或复审，不得直接各修各的。

2. 今后新增工具，必须先补：首要使用角色、次要使用角色、典型场景、默认结果落点、绝不能替代谁。  
3. 若某角色文件与本文件的工具优先级冲突，以本文件为准，先修角色文件，再修装配链。  
4. 当前工具层在治理 / 安全面 / 运行面已经补强，下一轮最应继续填补：
   - 高风险批量操作工具
   - 审计一致性工具
   - 运行面批量回放 / 事故重建工具

# 十、结论

到当前阶段，HGS 工具体系已经从“散装分析器”升级为：

- 真相判定工具
- 差异 / 漂移工具
- 运行面工具
- 体验审查工具
- 验证与治理工具
- 安全 / 门禁工具
- 沉淀工具

本总表的使命只有一个：

**让每个角色知道什么时候该先用哪个工具，工具结果该落到哪，工具不能替代谁。**
