> 版本：formal-2026-03-31-r1

> 说明：本文件是**工具治理文档**，不是运行入口；不替代 `MANIFEST` / `Master Loader`，但为后续接入装配链提供唯一工具调用依据。

# 一、总原则

## 1. 工具不是角色

HGS 中的工具 Skill 只做四件事：

1. 结构化分析  
2. 证据抽取  
3. 差异识别  
4. 结果格式化  

工具 **不能** 做四件事：

1. 替代 Owner 拍板真相  
2. 越权定义业务规则  
3. 宣布已闭环  
4. 绕过 P9 / P10 的裁决链  

一句话：

**工具负责“看清楚”，角色负责“拍板、执行、验证、收口”。**

## 2. 工具调用的基本顺序

```text
先识别真相 Owner / 执行 Owner
  ↓
再选对应工具做结构化分析
  ↓
工具结果进入 issue evidence / verdict / plan
  ↓
角色基于工具结果拍板或执行
```

## 3. 工具结果的默认落点

工具输出不得悬空，必须进入至少一个协议或角色输出：

- `ISSUE-LEDGER`
- `P8-EXEC-REPORT`
- `P9-REVIEW-VERDICT`
- `QA-VALIDATION-PLAN`
- `QA-VERIFICATION-RESULT`
- `BUSINESS-RULE-DECISION`
- `IDENTITY-DECISION`
- `BILLING-VERDICT`
- `RELEASE-CONFIG-VERDICT`
- `SECURITY-RISK-VERDICT`
- `CONTROL-PLANE-VERDICT`
- `EXECUTION-PLANE-VERDICT`
- `AGENT-OPS-VERDICT`
- `ENDUSER-SUPPORT-VERDICT`
- `DOCS-KNOWLEDGE-UPDATE`

# 四、按角色看的工具调用关系

## 1. Product / Business Rules Owner

### 优先工具
- `70_Business_Rule_Matrix_SKILL.md`
- `71_State_Machine_Consistency_SKILL.md`

### 什么时候必须用
- 规则口径不清
- 状态语义冲突
- 平台 / 代理 / 用户的动作边界不清
- 自助边界存在争议

### 输出落点
- `BUSINESS-RULE-DECISION`
- 必要时同步 `ISSUE-LEDGER`

### 禁止跳过的情况
- 同一规则被不同模块解释成不同语义时，不得只靠口头说明，必须至少调用一次矩阵或状态机工具。

---

## 2. Auth / Identity Owner

### 优先工具
- `72_JWT_Inspector_SKILL.md`
- `73_Device_Identity_Diff_SKILL.md`
- `74_Session_Refresh_Trace_SKILL.md`
- `88_Console_Auth_Flow_Trace_SKILL.md`
- `90_StepUp_Resume_Checker_SKILL.md`

### 什么时候必须用
- token 类型 / scope / exp 争议
- 换机 / 重装 / identity drift
- refresh 竞争、多标签会话异常
- Console 认证链、提权链异常

### 输出落点
- `IDENTITY-DECISION`
- 必要时同步 `ISSUE-LEDGER`

### 禁止跳过的情况
- 只凭 401 / 403 现象，不看 token / refresh / identity 证据时，不得直接下结论。

---

## 3. Billing / Entitlement Owner

### 优先工具
- `76_Billing_Ledger_Reconciler_SKILL.md`
- `77_Quota_Usage_Analyzer_SKILL.md`
- `78_Freeze_Reversal_Diagnoser_SKILL.md`

### 什么时候必须用
- 余额 / 冻结 / 可用额争议
- 名额占用 / 释放 / 配额争议
- 授权 / 续期失败后的冻结 / 冲正争议

### 输出落点
- `BILLING-VERDICT`
- 必要时同步 `AGENT-OPS-VERDICT` / `ENDUSER-SUPPORT-VERDICT`

### 禁止跳过的情况
- 点数、冻结、名额问题不得只看快照或口头描述，至少要有账本 / 配额 / 时间线工具证据之一。

---

## 4. Release / Config Owner

### 优先工具
- `101_Compatibility_Matrix_SKILL.md`

### 什么时候必须用
- 版本组合争议
- `version_min` / 强制升级 / 回滚门槛争议
- Console / Worker / 服务端版本边界不清

### 输出落点
- `RELEASE-CONFIG-VERDICT`
- 必要时同步 `ISSUE-LEDGER`

### 禁止跳过的情况
- 没有版本组合分析就做“升级 required / compatible / incompatible”结论。

---

## 5. Security / Risk Owner

### 可优先借用工具
- `72_JWT_Inspector_SKILL.md`
- `82_Network_Trace_Reviewer_SKILL.md`
- `90_StepUp_Resume_Checker_SKILL.md`
- `101_Compatibility_Matrix_SKILL.md`（涉及发布口子时）

### 什么时候必须用
- 越权路径疑似来自 token / scope / step-up 链
- 高风险动作门禁疑似只做了表面校验
- 请求上下文与权限表达不一致

### 输出落点
- `SECURITY-RISK-VERDICT`

### 说明
- 当前未单独配“安全专用工具”；安全角色优先借用身份、抓包、提权恢复、版本边界工具。

# 七、哪些场景必须至少调用一个工具

以下场景禁止纯靠口头判断，必须至少调用一个 Tool Skill：

1. **规则冲突**：至少调用 `70` 或 `71`  
2. **身份 / token / 刷新链争议**：至少调用 `72` / `74` / `88` / `90` 之一  
3. **名额 / 配额 / 点数 / 冻结争议**：至少调用 `76` / `77` / `78` 之一  
4. **接口联调争议**：至少调用 `79` 或 `82`  
5. **版本兼容争议**：至少调用 `101`  
6. **宣称已验证**：至少调用 `95` 或 `96` 之一  
7. **要沉淀成 SOP**：至少调用 `106`

# 八、工具越界红线

## 1. 不能替代真相角色

- `70` 不能替代 Product 拍板规则  
- `72/73/74/88/89/90` 不能替代 Auth / Control Plane / Console Runtime 拍板真相  
- `76/77/78` 不能替代 Billing 拍板账务真相  
- `101` 不能替代 Release / P10 拍板是否接受发布风险  

## 2. 不能替代验证结论

- `95` 不能等于“已经验证通过”  
- `96` 不能等于“已经全部测完”  

## 3. 不能替代文档治理

- `106` 不能把未拍板争议写进 SOP  
- `106` 不能替代 Docs Owner 做适用范围和受众把关  

## 4. 不能绕过 P9

- 工具发现多 owner 冲突、contract 漂移、状态漂移时，必须回到 P9 拆单或复审，不得直接各修各的。

# 九、当前最重要的执行建议

1. 后续把工具接入 `MANIFEST` / `Master Loader` 时，应以本文件作为唯一工具调用依据。  
2. 今后新增工具，必须先补：
   - 首要使用角色
   - 次要使用角色
   - 典型场景
   - 默认结果落点
   - 绝不能替代谁
3. 若某角色文件与本文件的工具优先级冲突，以本文件为准，先修角色文件，再修装配链。  
4. 今后补专属工具时，应优先填补当前空位最大的两个方向：
   - `Execution Plane / SRE` 的心跳 / trace / 运行异常工具  
   - `Security / Risk` 的高风险动作门禁与绕过分析工具

# 十、结论

到当前阶段，HGS 工具体系已经不再是“散装分析器”，而是形成了：

- 真相判定工具
- 差异 / 漂移工具
- 体验审查工具
- 验证工具
- 沉淀工具

本总表的使命只有一个：

**让每个角色知道什么时候该先用哪个工具，工具结果该落到哪，工具不能替代谁。**
