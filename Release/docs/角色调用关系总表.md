# HGS 角色调用关系总表

> 版本：formal-2026-03-31-r1  
> 目的：把 HGS 网络验证系统中各角色的调用顺序、拍板权限、协同边界、升级路径一次性钉死。  
> 说明：本文件是**角色调用治理文档**，不是运行入口，不替代 `MANIFEST` / `Master Loader`，但为后续接装配链提供唯一角色协同依据。

---

# 一、总原则

## 1. 角色调用的四层顺序

HGS 的角色协同，不按“谁声音大谁先来”，而按以下顺序：

```text
真相层 → 审查裁决层 → 执行层 → 验证/运行/沉淀层
```

对应到当前角色体系：

- **真相层**：P10、Product / Business Rules、Auth / Identity、Billing / Entitlement、Release / Config、Security / Risk、Control Plane、Execution Plane
- **审查裁决层**：P9 Principal
- **执行层**：P8 Backend、Frontend Logic、UI Surface、Console Runtime、Console Management Experience、LanrenJingling、Agent、EndUser、P8 Enhanced
- **验证/运行/沉淀层**：QA、SRE、Knowledge / Documentation

## 2. 一个问题进入系统后的默认问题

每个问题，必须依次回答：

1. **这件事的真相是谁说了算？**
2. **这件事应该由谁审查和拆单？**
3. **这件事应该由谁执行？**
4. **这件事应该由谁证明修好了？**
5. **这件事应该由谁沉淀下来，防止下次再靠人脑？**

## 3. 一条硬规则

**执行角色不能替代真相角色拍板；真相角色不能跳过 P9 直接裸派执行；验证角色不能替代真相角色重新定义问题；文档角色不能把未裁定争议写成正式规则。**

# 二、角色调用总流程

## 标准调用链

```text
P10（仅战略级初筛或冲突时）
  ↓
真相 Owner 识别（Product / Auth / Billing / Release / Security / Control Plane / Execution Plane / Agent Ops / EndUser Support）
  ↓
P9 Principal 审查、拆单、定 owner、定边界
  ↓
对应 P8 执行角色处理
  ↓
QA 验证 / SRE 运行观测 / 体验协议回收
  ↓
P9 复审
  ↓
P10 终审（仅必要时）
  ↓
Knowledge / Documentation 沉淀
```

## 什么时候不从 P10 开始

绝大多数问题**不需要**先找 P10。只有出现以下情况，才需要 P10 先入场：

- 路线冲突
- 优先级重排
- 高风险与推进速度之间要做战略权衡
- 会改变平台/代理/用户的利益或承诺

否则，默认从**真相 Owner 识别 + P9 审查**开始。

# 四、问题入口分流表

## 1. 规则不清 / 说法不一致

```text
先找：11 Product / Business Rules Owner
再交：20 P9 拆单
需要时协同：12 Auth / 13 Billing / 16 Agent Ops / 17 EndUser Support
```

## 2. Token / 会话 / 设备身份 / project scope

```text
先找：12 Auth / Identity Owner
若涉及运行时执行门禁：19 Execution Plane Owner
若涉及 Console 上下文消费：33A Console Runtime Engineer
再交：20 P9 拆执行项
```

## 3. 点数 / 冻结 / 冲正 / 权益 / 配额

```text
先找：13 Billing / Entitlement Owner
若涉及代理动作：16 Agent Operations Owner
再交：20 P9 → 31 Backend / 35 Agent
```

## 4. 后台显示正常但任务跑不起来

```text
先找：19 Execution Plane Owner
协同：18 Control Plane Owner、34 LanrenJingling、38 SRE
再交：20 P9 拆运行 / 控制两边 issue
```

## 5. 界面难用 / 看不懂 / 操作反馈差

```text
普通页面：先找 32A UI Surface Engineer
Console 管理台：先找 33B Console Management Experience Engineer
若表面问题背后是逻辑错：协同 32B / 33A
```

## 6. 状态对不上 / 路由上下文漂移 / 请求顺序乱

```text
普通前端：先找 32B Frontend Logic Engineer
Console：先找 33A Console Runtime Engineer
若 contract 漂移：再交 20 P9 / 31 Backend
```

## 7. 版本 / 配置 / 灰度 / 回滚

```text
先找：14 Release / Config Owner
协同：38 SRE、37 QA、19 Execution Plane、33A / 34
必要时升级：10 P10
```

## 8. 越权 / 高风险动作 / 安全门禁

```text
先找：15 Security / Risk Owner
协同：12 Auth、14 Release、31 Backend、32A / 33B、37 QA
必要时升级：10 P10
```

## 9. 代理总是绕路、不会闭环、SOP 混乱

```text
先找：16 Agent Operations Owner
逐单执行层：35 P8 Agent
若是系统信息缺口：11 / 13 / 39 协同
```

## 10. 用户总是看不懂、不会自助、只能找人

```text
先找：17 EndUser Support Owner
逐单执行层：36 P8 EndUser
若是规则/身份/点数真相问题：11 / 12 / 13 协同
```

## 11. 线上异常 / 日志追不到 / Trace 断了

```text
先找：38 SRE / Observability Owner
协同：19 Execution Plane、14 Release、31 / 34
再交：20 P9 拆工单
```

## 12. 这次解决了，但以后还会重复解释

```text
先找：39 Knowledge / Documentation Owner
协同：对应真相 Owner + P9 + QA + UI + 一线角色
```

# 五、谁能拍板、谁不能拍板

## 1. 真相拍板

只能由对应 Owner 拍板：

- 业务规则真相 → **11 Product / Business Rules Owner**
- 身份真相 → **12 Auth / Identity Owner**
- 账务与权益真相 → **13 Billing / Entitlement Owner**
- 发布与配置边界 → **14 Release / Config Owner**
- 安全风险边界 → **15 Security / Risk Owner**
- 代理运营 owner 边界 → **16 Agent Operations Owner**
- 用户支持 owner 边界 → **17 EndUser Support Owner**
- 控制平面边界 → **18 Control Plane Owner**
- 执行平面运行事实 → **19 Execution Plane Owner**

## 2. 不能拍板但能执行

以下角色主要负责执行，**不能重新发明真相**：

- 30 P8 Enhanced
- 31 P8 Backend
- 32A UI Surface Engineer
- 32B Frontend Logic Engineer
- 33A Console Runtime Engineer
- 33B Console Management Experience Engineer
- 34 P8 LanrenJingling
- 35 P8 Agent
- 36 P8 EndUser

## 3. 不能拍板但能裁决流程

- **20 P9 Principal**：能拆单、定 owner、定边界、复审，但不能越权重写真相
- **37 QA**：能判断验证是否足够，但不能重写真相
- **38 SRE**：能判断线上归因与止血建议，但不能重写真相
- **39 Docs**：能沉淀规则，但不能先于真相角色拍板

# 六、必须协同、不能单飞的典型组合

## 1. Product + Auth

适用于：
- 换机 / 重装 / 解绑 / 自助恢复边界
- 身份问题背后其实是业务规则问题

## 2. Product + Billing

适用于：
- 扣点该不该发生
- 哪类恢复是否免费
- 套餐 / 配额 / 权益与经营口径冲突

## 3. Control Plane + Execution Plane

适用于：
- 后台显示正常但执行跑不起来
- 配置允许与运行事实不一致
- 授权存在但执行门禁失败

## 4. UI Surface + Frontend Logic

适用于：
- 页面看起来有问题，但根因在逻辑层
- 逻辑修完后，表面反馈仍然不清楚

## 5. Console Runtime + Console Management Experience

适用于：
- 管理台“能点但不好管”
- 操作失败到底是上下文丢了，还是界面没讲清楚

## 6. Release + SRE + QA

适用于：
- 要不要灰度/回滚
- 版本组合争议
- 线上异常和发布窗口相关

## 7. Agent Ops + P8 Agent

适用于：
- 代理 owner 层 SOP 不清
- 一线代理执行层逐单闭环

## 8. EndUser Support + P8 EndUser

适用于：
- 用户支持 owner 路径不清
- 一线用户排查和快速恢复

# 七、升级规则总表

## 升级到 P9 的条件

满足任一项，必须由 P9 介入：

- owner 不清
- 多个 owner 交叉依赖
- 需要拆 issue
- 需要定最大改动边界
- 需要复审 / reopen / escalate

## 升级到 P10 的条件

满足任一项，必须升级：

- 路线冲突
- 风险容忍度需要战略拍板
- 优先级需要重排
- 会改变平台 / 代理 / 用户利益或承诺
- 发布 / 安全 / 经营策略需要路线级判断

## 升级给用户的条件

只有内部穷尽后才允许升级问用户：

- 需要用户业务取舍
- 需要用户授权高风险动作
- 需要真实外部动作（例如真实外发、真实账户操作）
- P9 / P10 已确认内部不存在可接受默认路径

# 八、角色调用口诀

## 总口诀

```text
先找真相 Owner，
再交 P9 裁单，
再由 P8 执行，
再让 QA / SRE / 体验证明，
最后交 Docs 沉淀。
```

## 边界口诀

```text
Owner 负责“谁说了算”，
P9 负责“怎么拆和怎么复审”，
P8 负责“怎么做出来”，
QA/SRE 负责“怎么证明与怎么观测”，
Docs 负责“怎么留下来”。
```

## 前端口诀

```text
UI Surface 管“看起来和用起来”，
Frontend Logic 管“状态、上下文、请求、竞态”。
```

## Console 口诀

```text
Console Runtime 管“管理入口能不能正确运行”，
Console Management Experience 管“管理者能不能看懂、敢点、少误操作”。
```

## 平面口诀

```text
Control Plane 管后台真相，
Execution Plane 管运行事实。
```

# 十、结论

到当前阶段，HGS 角色体系已经从“按岗位名堆角色”，进入到“按真相、裁决、执行、验证、运行、沉淀”分层协同。  
本总表的使命只有一个：

**让每个问题进来后，都知道该先找谁、谁能拍板、谁必须协同、谁不能越权。**
